# protocol.md

# WebSocket Communication Protocol (HTTP‑like RPC + Event)

Version: v1

Goal: Provide a unified communication protocol so WebSocket behaves
similar to HTTP while supporting realtime event push.

------------------------------------------------------------------------

# 1. Design Principles

1.  HTTP-like mental model
2.  Unified message envelope
3.  Request / Response RPC
4.  Server Event Push
5.  Topic subscription
6.  Versioned protocol
7.  Type-safe DTO support

Core idea:

WebSocket = HTTP RPC + Event Push

------------------------------------------------------------------------

# 2. Unified Message Envelope

All WebSocket messages must follow this structure.

``` ts
interface WsMessage<T = any> {
  v: number
  id?: string
  type: 'req' | 'res' | 'event'
  route: string
  code?: number
  msg?: string
  data?: T
  ts?: number
}
```

Field description:

  field   description
  ------- -------------------
  v       protocol version
  id      request id
  type    req / res / event
  route   business route
  code    response code
  msg     error message
  data    payload
  ts      server timestamp

------------------------------------------------------------------------

# 3. Request

Client → Server

``` json
{
  "v":1,
  "id":"a1b2",
  "type":"req",
  "route":"user.profile",
  "data":{}
}
```

------------------------------------------------------------------------

# 4. Response

Server → Client

``` json
{
  "v":1,
  "id":"a1b2",
  "type":"res",
  "route":"user.profile",
  "code":0,
  "data":{
    "userId":1,
    "name":"Tom"
  }
}
```

------------------------------------------------------------------------

# 5. Event Push

Server → Client

``` json
{
  "v":1,
  "type":"event",
  "route":"bag.update",
  "data":{
    "items":[]
  }
}
```

------------------------------------------------------------------------

# 6. Route Naming

Format:

domain.action

Examples:

user.profile\
user.login\
bag.items\
bag.update\
game.status

Rules:

-   lowercase
-   dot separated
-   no HTTP method

------------------------------------------------------------------------

# 7. Error Codes

  code   meaning
  ------ ----------------
  0      success
  400    bad request
  401    unauthorized
  403    forbidden
  404    not found
  500    internal error

Example:

``` json
{
  "id":"a1b2",
  "type":"res",
  "route":"user.profile",
  "code":401,
  "msg":"unauthorized"
}
```

------------------------------------------------------------------------

# 8. Topic Subscription

Client subscribe:

``` json
{
  "type":"req",
  "route":"topic.subscribe",
  "data":["bag","status"]
}
```

Client unsubscribe:

``` json
{
  "type":"req",
  "route":"topic.unsubscribe",
  "data":["bag"]
}
```

Server implementation example:

socket.join(topic)

Push:

io.to(topic).emit(...)

------------------------------------------------------------------------

# 9. Timestamp

Field:

ts

Rules:

-   generated by server
-   unix seconds

Usage:

-   debugging
-   log tracing
-   deduplication

------------------------------------------------------------------------

# 10. Request ID

Field:

id

Rules:

-   unique per request
-   uuid or random string
-   used for promise mapping

------------------------------------------------------------------------

# 11. HTTP Mapping

  HTTP                WS
  ------------------- --------------
  GET /user/profile   user.profile
  POST /user/login    user.login
  GET /bag/items      bag.items
  push update         bag.update

------------------------------------------------------------------------

# 12. Client SDK Design

Recommended client API:

    ws.request(route,data)
    ws.subscribe(topics)
    ws.unsubscribe(topics)
    ws.on(event,handler)

Example:

    await ws.request('user.profile')

    ws.subscribe(['bag'])

    ws.on('bag.update',data=>{
      store.update(data)
    })

------------------------------------------------------------------------

# 13. Server Structure

Recommended structure:

    ws/
      gateway.ts
      router.ts
      topics.ts
      emit.ts
      deduplicator.ts

Responsibilities:

gateway --- socket connection\
router --- route handlers\
topics --- topic management\
emit --- push wrapper\
deduplicator --- prevent duplicate push

------------------------------------------------------------------------

# 14. Push Deduplication

Avoid repeated pushes when data unchanged.

Example strategy:

hash(data)

if same as last push → skip

Useful for:

status updates\
dashboards\
game state

------------------------------------------------------------------------

# 15. Versioning

Protocol includes field:

v

Strategy:

v1 initial version\
v2 backward compatible additions\
v3 breaking changes

------------------------------------------------------------------------
